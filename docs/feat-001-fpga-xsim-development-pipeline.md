## Goal

Bring NeuronFabric transformer training onto Xilinx FPGA in two phases:

**Phase 1 вҖ” Correctness proof (XSim simulation)**
Sequential RTL using shared compute units. Proves every arithmetic operation matches the C# reference within tolerance. No resource optimisation. All steps verified in Vivado XSim before any hardware is touched.

**Phase 2 вҖ” Parallel FPGA architecture (pre-hardware synthesis)**
Rewrite integrators (`attention_core`, `mlp_core`) so each weight matrix has its own co-located BRAM + DSP48 MAC array вҖ” all firing in the same clock cycle. This is the NeuronFabric paper architecture: local weights grouped around computing cores, replicated until BRAM/DSP48 budget is exhausted. Phase 2 begins after all Phase 1 XSim steps pass.

## Development order

### 1. BF16 arithmetic module
- BF16 multiply-accumulate (MAC) unit
- Verify: simulate a handful of known multiply-add pairs, compare to IEEE 754 BF16 reference
- Parameterise: latency (pipeline depth), number of parallel MACs

### 1b. Exp LUT module
- 256-entry BRAM ROM storing `2^(i/255)` for i=0..255 (matches `ExpLutHelper.Get(256)` exactly)
- Operation: `exp(x) = 2^n В· 2^f` where `n = floor(xВ·logвӮӮe)` (integer вҶ’ set FP32 exponent field), `f = frac` (вҶ’ BRAM lookup + linear interpolation between adjacent entries)
- NaN input вҶ’ output 0.0 (handles fully-masked causal attention rows)
- BRAM init file generated by C# `ExpLutHelper.Get(256)` вҶ’ `exp_lut_init.hex` вҶ’ loaded via `$readmemh`
- Verify: 32 test vectors covering negative, zero, positive, large negative (underflow вҶ’ 0), NaN вҖ” compare to `ExpLutHelper.Exp()` C# reference within BF16 rounding tolerance
- Parameterise: `LUT_SIZE` (256 or 512), pipeline depth

### 2. Matrix multiplication (BF16Г—BF16)
- Tiled MatMul built from BF16 MACs: C[MГ—N] = A[MГ—K] В· B[KГ—N], both operands BF16
- KГ—N parallel `bf16_mac` units (c_fp32=0: pure product) + FP32 adder tree
- K=4 adder tree: `(p0+p1) + (p2+p3)` вҖ” 2 pipeline stages; total latency = MAC_LATENCY + 2
- Verify: 4Г—4Г—4 random BF16 matrices, C# reference uses **ReferenceExactHardwareMode** (`(float)((double)x op (double)y)` per operation, matching XSim shortrealвҶ’double promotion) вҖ” all 16 elements within **1 ULP**
- Note: models **both operands BF16** вҖ” useful for fully-quantized inference but NOT the bf16w training path (activations are always FP32)

### 2b. Matrix multiplication (FP32Г—BF16 вҖ” bf16w training path)
- Same adder tree as step 2, but `a_fp32 [31:0]` replaces `a_bf16 [15:0]`
- Maps to `AdamBF16WeightsAttentionCore` forward pass: weights stored BF16, decoded on-the-fly; activations always FP32
- `bf16_mac` needs a new `FP32Г—BF16` variant (or mixed-input port): decode B from BF16, multiply as FP32Г—FP32
- Verify: A = FP32 activations, B = BF16 weights; C# reference is **ReferenceExactHardwareMode** (adder-tree structure, each op as `(float)((double)x op (double)y)`) вҖ” all 16 elements within **1 ULP**
- This is the module wired directly into step 3 (attention QKV projections, output projection) and step 4 (FF layers)

> вҡ пёҸ **IMPORTANT вҖ” XSim `shortreal` uses x87 80-bit intermediate precision (Windows)**
>
> The current RTL uses `shortreal` variables for all arithmetic. On Windows, XSim evaluates `shortreal` expressions using **x87 80-bit extended precision** internally, then rounds to 32-bit via `$shortrealtobits()`. This is **not** strict IEEE 754 single-precision, and is **not** double-promotion (earlier versions of this doc incorrectly stated double-promotion).
>
> **Empirically measured (tb_fp32_add_tree, seed 42):** worst-case deviation from C# native float32 is **6 ULP**. All testbenches for purely-combinatorial modules use **8 ULP tolerance** to accommodate this.
>
> **Synthesized FPGA**: `shortreal` gets inferred by Vivado into the **Floating-Point Operator IP** (DSP48 + LUT/FF), which uses strict IEEE 754 single-precision. The x87 deviation is a **simulation artifact only** вҖ” synthesized results will match C# float32 exactly.
>
> **The real risk is verification**: testbenches verified under 8 ULP XSim tolerance may mask 1-2 ULP differences in the synthesized IP. A post-synthesis verification pass will need a stricter reference matched to the synthesized IP's rounding.
>
> **Action required before hardware synthesis** (not before XSim steps 3вҖ“7):
> Replace `shortreal` with explicit IEEE 754 RTL or instantiated Xilinx FP Operator IP, and update C# reference to match.

### 3. Attention sub-modules (modular build-up before integration)

Step 3 is broken into independent sub-modules, each verified standalone before wiring into `attention_core.sv`.

#### 3a. FP32Г—FP32 matrix multiply (`fp32_matmul.sv`)
- C[MГ—N] = A[MГ—K] В· B[KГ—N], both operands FP32 activations (no weights)
- Used for: QГ—KбөҖ score matrix (TГ—d Г— dГ—T), attention-weighted sum (TГ—T Г— TГ—d)
- Architecture: KГ—N `shortreal` multiply units + 2-stage FP32 adder tree (identical structure to `bf16w_matmul`)
- Verify: 4Г—4Г—4 random FP32 matrices, C# **ReferenceExactHardwareMode** вҖ” all 16 elements within 1 ULP

#### 3b. FP32 softmax row (`softmax.sv`)
- Input: T FP32 scores; output: T FP32 probabilities
- Causal mask applied before exp: scores at masked positions set to вҲ’вҲһ вҶ’ exp вҶ’ 0
- Sub-modules instantiated:
  - `exp_lut.sv` вң… вҖ” per-element exp
  - `fp32_add_tree.sv` вң… вҖ” reduction sum over T elements (parameterised depth)
  - `fp32_div.sv` вң… вҖ” scalar FP32 divide for normalisation (1/sum Г— each exp output)
- Verify: T=4 row, causal mask, compare to C# `AttentionCore` softmax reference

#### 3c. Attention core (`attention_core.sv`) вҖ” integration
- Wires: `bf16w_matmul` (QKV + output proj) + `fp32_matmul` (scores + weighted sum) + `softmax`
- Verify: forward pass output matches `AttentionCore.Forward()` C# reference for small config (heads=1, d=4, T=4)
- **вҡ пёҸ Current implementation is sequential** (one `bf16w_matmul` instance reused for Q, K, V, Wo). This minimises resources for XSim verification but is not the target architecture.
- **Target parallel architecture** (required before hardware synthesis):
  - 3Г— `bf16w_matmul` instances for Q, K, V firing simultaneously
  - 2Г— `fp32_matmul` instances for scores and AV (can overlap after Q/K/V done)
  - Parallelising Q/K/V alone gives 3Г— throughput on the projection phase
  - This is the natural FPGA mapping: each matmul unit occupies a DSP48+BRAM region, all fire in the same clock cycle

### 4. MLP (feed-forward block)

The feedforward block from `AttentionLayer.Forward`:
```
H1 = X В· Wff1        [TГ—d] Г— [dГ—ffDim]  вҶ’ [TГ—ffDim]   (bf16w matmul)
G  = GeLU(H1)        [TГ—ffDim]           вҶ’ [TГ—ffDim]   (element-wise)
Y  = G В· Wff2        [TГ—ffDim] Г— [ffDimГ—d] вҶ’ [TГ—d]     (bf16w matmul)
```

GeLU formula (tanh approximation, matching C# `AttentionLayer.Gelu`):
```
GeLU(x) = 0.5 В· x В· (1 + tanh(вҲҡ(2/ПҖ) В· (x + 0.044715В·xВі)))
        = 0.5 В· x В· (1 + tanh(0.7978845608 В· (x + 0.044715В·xВі)))
```

| Module | Purpose |
|---|---|
| `gelu.sv` | Element-wise GeLU: polynomial + tanh approximation, pipelined | вң… done |
| `mlp_core.sv` | Integrator: bf16w_matmul вҶ’ gelu вҶ’ bf16w_matmul | вң… |

#### Sub-steps

**4a. `gelu.sv`**
- Input/output: FP32 scalar (or row of T FP32 values)
- Tanh approximation: `0.5В·xВ·(1+tanh(cВ·(x+0.044715В·xВі)))` where `c=0.7978845608`
- Implementation options (decide at this stage):
  - **LUT-256 for tanh**: `tanh(x) = 2В·Пғ(2x)вҲ’1`; map through a LUT over the relevant range вҖ” consistent with the exp LUT approach already used in softmax
  - **Polynomial**: degree-5 minimax over [вҲ’3, 3], clamp outside вҖ” simpler RTL but less principled
- Verify: T=4 random FP32 inputs, compare to C# `AttentionLayer.Gelu(x)` with `ReferenceExactHardwareMode`; expected tolerance вүӨ2 ULP (single multiply+tanh stage, same x87 artifact as softmax)

**4b. `mlp_core.sv`**
- Interface mirrors `attention_core.sv`: `x_wr_*` (FP32 activations), `wff1_wr_*` / `wff2_wr_*` (BF16 weights), `start`, `out_row`, `out_valid`, `out_row_idx`
- Reuses one `bf16w_matmul` instance sequentially for both Wff1 and Wff2 passes (same area-saving strategy as `attention_core`)
- FSM: LD_W1 вҶ’ RUN_W1 вҶ’ WAIT_W1 вҶ’ GELU вҶ’ LD_W2 вҶ’ RUN_W2 вҶ’ WAIT_W2 вҶ’ OUTPUT вҶ’ IDLE
- Parameters: T, D (=embedDim), FF (=ffDim), MAC_LAT, MUL_LAT
- Verify: forward output matches `AttentionLayer` FF sub-block (`xNorm2 вҶ’ Wff1 вҶ’ GeLU вҶ’ Wff2`) for T=4, D=4, FF=16 вҖ” all TГ—D elements within `XSimCollection.UlpRelTol`

### 5. LayerNorm

Forward (per token row, C# `LayerNorm.Forward`):
```
Ој      = (1/D) В· ОЈ x[d]
ПғВІ     = (1/D) В· ОЈ (x[d] вҲ’ Ој)ВІ
invStd = 1 / вҲҡ(ПғВІ + Оө)          Оө = 1e-5
y[d]   = Оі[d] В· (x[d] вҲ’ Ој) В· invStd + ОІ[d]
```

One `layernorm.sv` processes one token row (D FP32 values) at a time; the caller feeds T rows sequentially.

No new arithmetic primitives are needed for mean/variance accumulation or scale вҖ” those are `shortreal` add/multiply, same as all other modules. However, `invStd = 1/вҲҡ(ПғВІ+Оө)` requires a **synthesizable sqrt** вҖ” `$sqrt()` is simulation-only and does NOT synthesize. We implement `fp32_sqrt.sv` using iterative NewtonвҖ“Raphson (same approach as `fp32_div.sv`), both for synthesis correctness and so the Phase 1 test exercises the real algorithm.

C# reference for `fp32_sqrt.sv`: add `ExpLutHelper.RecipSqrt(float x)` вҖ” same seed table + same two iterations using `(float)((double)a op (double)b)` per multiply (`ReferenceExactHardwareMode`). `LayerNorm.Forward` uses `1f / MathF.Sqrt(var_ + _eps)` вҖ” the C# vecgen will call `ExpLutHelper.RecipSqrt(var_ + eps)` instead. Also add a pure-C# unit test that `RecipSqrt(x)` converges to within `VsSoftwareRelTol` of `1f / MathF.Sqrt(x)` over a range of positive inputs.

| Module | Purpose | Status |
|---|---|---|
| `fp32_sqrt.sv` | `1/вҲҡx` via NewtonвҖ“Raphson, combinatorial, matches synthesized FP | вң… done |
| `layernorm.sv` | Mean + variance reduction, scale/shift; instantiates `fp32_sqrt` | вң… done |

#### Sub-steps

**5a. `fp32_sqrt.sv`**
- Computes `1/вҲҡx` (reciprocal square root) for FP32 scalar input
- Implementation: seed from top 8 bits of mantissa (256-entry ROM), then 2Г— NewtonвҖ“Raphson: `r вҶҗ r В· (1.5 вҲ’ 0.5 В· x В· rВІ)` using `shortreal` multiply вҖ” each iteration is one `shortreal` assignment (one rounding)
- Fully synthesizable: only `shortreal` multiply + ROM вҖ” no `$sqrt()`
- C# reference: add `ExpLutHelper.RecipSqrt(float x)` вҖ” same seed table + same two iterations using `(float)((double)a op (double)b)` per multiply (`ReferenceExactHardwareMode`)
- Pure-C# unit test (`Neuro.Attention.Tests`): `RecipSqrt(x)` converges to within `VsSoftwareRelTol` of `1f / MathF.Sqrt(x)` over a range of positive inputs
- XSim verify: 32 random positive FP32 inputs; compare RTL output to `ExpLutHelper.RecipSqrt()` within `VsSoftwareRelTol`

**5b. `layernorm.sv`**
- Interface: `x_in [DГ—32]` (FP32 row), `gamma [DГ—32]`, `beta [DГ—32]`, `start`, `y_out [DГ—32]`, `out_valid`
- Parameters: `D` (embed dim), `MUL_LAT`, `ADD_LAT`
- FSM:
  - `SUM_X` вҖ” accumulate ОЈx[d] over D cycles using running FP32 adder
  - `MEAN` вҖ” divide by D (multiply by `1/D`, precomputed constant)
  - `SUM_VAR` вҖ” second pass: accumulate ОЈ(x[d]вҲ’Ој)ВІ вҖ” requires buffering x (D-deep shift register)
  - `VAR` вҖ” divide by D вҶ’ ПғВІ; then feed `ПғВІ+Оө` to `fp32_sqrt` instance (combinatorial вҶ’ result available same cycle)
  - `SCALE` вҖ” third pass: `y[d] = Оі[d] В· (x[d] вҲ’ Ој) В· invStd + ОІ[d]` вҖ” one multiply chain per cycle
  - `OUTPUT` вҖ” assert `out_valid`, stream T rows from caller
- Buffering: x[] stored in a D-entry FP32 shift register (or BRAM for large D); Оі[] and ОІ[] loaded once, held in registers
- Verify: T=4 token rows, D=4, random FP32 inputs and random Оі/ОІ weights; C# `LayerNorm.Forward()` reference; all TГ—D elements within `VsSoftwareRelTol`

### 6. Adam optimiser

Implements the BF16-weights Adam update step, matching `AdamBF16WeightsAttentionCore.ApplyUpdate()` exactly. Every C# arithmetic operation is mapped to a synthesizable RTL equivalent вҖ” no simulation-only primitives, no deferred work for Phase 2.

Per-parameter layout (matches the C# `_state` dictionary):
```
w     вҖ” BF16  (2 bytes) вҖ” current weight, quantised each step
m     вҖ” FP32  (4 bytes) вҖ” first moment (ОІвӮҒ EMA of gradient)
v     вҖ” FP32  (4 bytes) вҖ” second moment (ОІвӮӮ EMA of gradientВІ)
```

Update rule (one scalar, one clock cycle in RTL):
```
m    вҶҗ ОІвӮҒВ·m + (1вҲ’ОІвӮҒ)В·g
v    вҶҗ ОІвӮӮВ·v + (1вҲ’ОІвӮӮ)В·gВІ
mHat = m / bc1                    -- fp32_div.sv
vHat = v / bc2                    -- fp32_div.sv
sqrtV = vHat * fp32_sqrt(vHat)    -- fp32_sqrt gives 1/вҲҡvHat; multiply back to get вҲҡvHat
denom = sqrtV + Оө                 -- shortreal add
wf   = Decode(w_bf16)             -- bit-shift (bf16_mac decode path)
wf  -= lr * mHat / denom          -- fp32_div.sv
w_bf16 вҶҗ Encode(wf)              -- truncate mantissa bits [15:0]
```

Constants: ОІвӮҒ=0.9, ОІвӮӮ=0.999, Оө=1e-8. `t` = global step counter (bias correction).

#### Sub-steps

**6a. `adam_cell.sv`** вҖ” scalar Adam update for one parameter
- Inputs: `g_fp32`, `w_bf16`, `m_fp32`, `v_fp32`, `lr_fp32`, `bc1_fp32`, `bc2_fp32`, `en`
- Outputs: `w_bf16_out`, `m_fp32_out`, `v_fp32_out` (registered, available next cycle)

- `bc1`, `bc2` are per-step FP32 inputs вҖ” caller (`adam_core`) computes `1 вҲ’ ОІвӮҒбө—`, `1 вҲ’ ОІвӮӮбө—` and passes them in
- Parameters: ОІвӮҒ, ОІвӮӮ, Оө hardcoded as localparams matching C# values
- Verify: 10 sequential Adam steps on one parameter; compare w_bf16, m, v after each step to C# `AdamBF16WeightsAttentionCore` reference within `VsSoftwareRelTol`

**6b. `adam_core.sv`** вҖ” applies `adam_cell` to an entire weight matrix

- Outputs: `w_bf16_out`, `m_out`, `v_out` (all RГ—C values, valid when `done=1`)
- FSM: IDLE вҶ’ UPDATE (RГ—C cycles, one `adam_cell` instance, one parameter per cycle) вҶ’ DONE
- Parameters: `R`, `C`
- Verify: Wq matrix (4Г—4), 100 Adam steps; compare final w_bf16/m/v to C# reference within `VsSoftwareRelTol`

## Cross-repo test flow

The two repos stay fully independent вҖ” no code dependency, just file conventions.

A new project **`Neuro.Attention.XSim.LocalTests`** (separate from `Neuro.Attention.Tests`) owns the FPGA test loop. It is never run in CI вҖ” local developer machine only.

```
neuro-fabric (C#)
  Neuro.Attention.XSim.LocalTests
    FpgaVecGen
      вҶ“ writes  run/fpga-testvecs/<module>/input.hex
                run/fpga-testvecs/<module>/expected.hex
    FpgaXSimRunner
      вҶ“ Process.Start("xvlog", "bf16_mac.sv tb_bf16_mac.sv")
      вҶ“ Process.Start("xelab", "tb_bf16_mac")
      вҶ“ Process.Start("xsim",  "tb_bf16_mac -runall")
      вҶ“ writes  run/fpga-testvecs/<module>/pass_fail.txt
    FpgaVecCheck
      вҶ“ reads   pass_fail.txt
      вҶ“ fails / passes as a normal xUnit test
```

- **Developer workflow is pure C#** вҖ” just `dotnet test Neuro.Attention.XSim.LocalTests`; the `xvlog`/`xelab`/`xsim` toolchain is invoked automatically behind the scenes, no manual shell commands needed
- **CI exclusion:** `Neuro.Attention.XSim.LocalTests` is not listed in `.github/workflows/build.yml` вҖ” the yaml runs only `Neuro.Attention.Tests` and `Neuro.Core.Tests`; XSim tests never execute in GitHub Actions
- Verilog source paths passed to `xvlog` resolve via env var `NEURO_FPGA_SRC` pointing to `neuro-fabric-fpga/rtl/` вҖ” the only coupling between the two repos
- `NEURO_TESTVECS` env var points to `neuro-fabric/run/fpga-testvecs/` for the hex file exchange folder
